Details¶

EzMojo intends to define a minimal OSGI based distributed OSC UDP packet aware nodal architecture (mainly, not restrictive).

The idea is to have a service discovery powered topology of OSC aware nodes. It consists in having OSC agents accessible thanks to UDP service , exposed with given support of EzMojo protocol described below.

The power of this solution is that it enables both relatively easy synchro facilities along with high compatibility with existing OSC solution, especially in the domain of music/VJing/DJing.

The solution will handle an image of OSC data struture as a tree, possibly handling several domains, subdomains. As a result it will ease the recursive use of it, with a possibly very efficient distributed query infrastructure and a decent multi point distributed event/message emitting/updating/binding.

For instance, lets deal with idiotic examples : "/foo/bar/buz", "/foo/bar/bee", "/mini/me", "zip/it"

Tree representation is direct : 

                            .
                           / \
                          /   \
                         /host1\
                        /_______\
                       /    |    \
                      foo  mini  zip
                     /      |      \
                   bar      me      it
                  /   \
                buz   bee

Following features will have to be supported :

  • "@" BANG @/foo/bar/bee
  • "" INFO this is raw text to be displayed
  • "(" FUNCTION (/foo/bar/bee ENVELOPPE 34 54 67 78
  • ":" GET :/foo/bar/bee
  • "+" SET +/foo/bar/bee f:1.0 b:true i:45 s:grr s:'ui' s:"kk"
  • "/" LEGACY_SET /foo/bar/bee f:1.0 b:true i:45 s:grr s:'ui' s:"kk"
  • "-" UNSET -/foo/bar/bee
  • "!" LISTEN !/mini/me
  • "#" UNLISTEN #/mini/me
  • "?" QUERY ?/foo/
  • "=" BIND =/zip/it, 192.168.1.100:1970
  • "$" SERVICE $/mini, 192.168.1.100:1970
  • ">" FORWARD >/zip/it, 192.168.1.100:1970
  • "<" UNFORWARD </zip/it
  • "&" CLOCK &/mini/me 2000
  • ")" TOGGLE )/mini/me
  • "." SELECT ./mini/me

Given the fact this architecture rely on DNS-SD service discovery it is clear that it must leverage local network stuff but can be accessible thanks to simple UDP connection to resolvable ports.

DNS-SD will then expose services expressed as OSC bundles. This way it will be easy to reuse information even with non EzMojo aware application only by using plain OSC.

Additional legacy and cross compatibilities must be envisionned such as osc2dmx, osc2midi and others possible (artnet, blinkm, living colors, not exhaustive ...).

Obviously, additional compatible layers must be contributables and be accessible for selection given an optional selector in the command.

for instance :

+" SET +/foo/bar/bee>>dmx, f:1.0, b:true, i:45, s:grr, s:'ui', s:"kk"

Previous expression will be translated to DMX (TBD ...)

A typical exchange between nodes and/or 3rd party applications could be expressed as:

A B

$/mini/me ----------> ...

<---------- $/mini/me, i:, s:

Infrastructure¶

In order to get the distribution leveraged in an efficient way, nodes have to be uniquely identified and offering some global facilities such as listing, discovery, ...

In order to get the nodes identified, the ezmojo protocol should be embedded into higher level frame given a domain name as identifier. This way it will be easier for users to get informations in a readable way.

For instance, three hosts respectively hosting an OSC node could get described like :

  • host1 : 192.168.1.100:4444 on host1
  • host2 : 192.168.1.101:4444 on host2
  • host3 : 192.168.1.102:4444 on host3

Each node could be controlled in an ezmojo way thanks to command syntax like :

  • host1, +/mini/me, i:1 s:'toto'
  • host, +/mini/me, i:2 s:'tata'
  • 192.168.1.102:4444, +/mini/me, i:2 s:'tata'

Some simple regex could be usefull to discriminate all or subset of host given their alias or IP@:port address.

Commands¶

An EzMojo facility common standard set could be :

  • %/list
  • %/reset
  • %/close
  • %/status
  • %/bind
  • %/deploy
  • %/remove
  • %/help
  • %/protocol

Some reserved domains must be define such as :

  • all (All accessible nodes)
  • here (All locals)

It could be possible to access specificly defined : (%aliases%)+, (%IP@:port)+

Protocols¶

Protocol is stupidly simple while a single character caracterize the kind of command and is followed by the domain it targets. Other stuff is pure OSC.

We need to distinguish two forms : textual and raw.

EzMojo (raw)¶

Raw EzMojo form aims at being easily transportable in terms of network facilities.

The basic frame is like 

    .----------------------------------------------------------------.
    |         |     |             |                                  |
    | TARGETS | CMD | OSC ADDRESS | LEGACY OSC STREAM (INCL ADDRESS) |
    |         |     |             |                                  |
    :----------------------------------------------------------------:

Textual EzMojo aims at expressing in an human readble form all commands. Here are some exampes :

  • "-" UNSET -/foo/bar/bee
  • "!" LISTEN !/mini/me
  • "#" UNLISTEN #/mini/me
  • "?" QUERY ?/foo/

BANG¶

It is usefull to discriminate what will cause a bang. Set commands are by nature triggering stuff while something actually changed.

However, while set commands embbed arguments, it is very unlikely that it is optimal for basic notifcations.

So the bang has been created. Bang command just deal with OSC address with no arguments. Obviously a bang does reset associated

GET¶

Simple getter functionnality sending typical legacy OSC messages back

SET¶

Simple OSC messages like : /foo/bar 1 "sss" 3.0 Human writability of OSC expressions just starting give two options :

  • /foo/bar (Added w/ TTL by default)
  • +/foo/bar (Added ad vitam eternam)

Arguments processing:

  • Args as textual expression
  • Args as binary OSC dump for networking purposes

UNSET¶

The unset command handles the fact to remove subtrees from a given domain or address regex expression. It obviously remove in the same time all listeners.

We will have to consider the BIND problematic later on.

LISTEN¶

Listen, basicallly registers listeners to a given EzMojo Node from its IP an port definition. It may also be introduced a notion of alias in order to compress the expressivity of addressing.

Listener just behave like a basic feedback. It appears to be a bit related to FORWARD concept in terms of APIs. However, it represents a listener concept where the equivalent would be an unlogical reverse forwarding ...

UNLISTEN¶

This holds the removal responsability of any corresponding registred listener according to a particular related domain or a regex based address filter.

QUERY¶

NOT SUPPORTED YET

Queries would have to be a kind of XPath and/or regex compatible giving possibility to retrieve an unordered OSC bundle with matching dommains and addresses. A particular query would be the usage of '' to get any completions of matching addresses. It can be extended to an XPath like query syntax

BIND¶

NOT SUPPORTED YET

Binding is the expression of synchronization on related messages between several nodes. In particular, several different nodes would like to share domains or addresses states accros the network for synchronization purpose. However this synchronization do not seem correlated to the notion of LISTENER and FORWARD in the sense it could probably not trigger LISTEN/ FORWARD event while not explicitally required by the internal BIND system.

This internal capapbility will have to coexist separately with other commands.

SERVICE¶

SERVICE is a simple query starting w/ '$' applied to a given domain and/or regex address filtering. The result of such querying returns a bundle of all services for the given node. While service holds a notion of composition, services could be extended to any domain, address and EzMojo subtree. Typical bundle result could describe a typical tree introspection and/or be enriched/filtered by a mechanism no specified yet.

FORWARD¶

NOT SUPPORTED YET

Forwarding holds responsability to strictly resend OSC information with a possible arguments modification (To be specified later on). FORWARD do not acts as LISTEN while LISTEN just behave the opposite way.

UNFORWARD¶

NOT SUPPORTED YET

Holds the removal responsability of any corresponding registred FORWARD according to a particular related domain or a regex based address filter.

SELECT¶

Set a given domain/subdomain as a root of all following commands.

For instance, selecting /mini/me will cause +/rocks command to be applied on the /mini/me node.

CLOCK¶

NOT SUPPORTED YET

Set a given domain/subdomain a bang clocking rate so that it is possible to automate a cyclic stimulation on a node or sub domain.

FUNCTION (KINETICS)¶

NOT SUPPORTED YET

Lets discriminate at least two categories : raw and additive. Kinetic raw action is definitly a kind of temporal scripting facility on top of the other ones.

Newly added functions are runned on the next bang until a eventual toggle.

RAW¶

Means that registering RAW stuff on a domain will trigger automatic message emmiting with at given rate, sampling, harmonics. (TBD)

  • TTL
  • pulse
  • loop
  • at
  • whenever
  • trigger
  • envelope

ADDITIVE¶

Means that registering an ADDITIVE stuff on a domain will trigger automatic message emmiting with adding corresponding values at given rate, sampling, harmonics at the and of argument arrays. (TBD)

  • SQR
  • SIN
  • OSC
  • SAW
  • ROT
  • ENVELOPPE

TOGGLE¶

NOT SUPPORTED YET

Enable/Disable an EzMojo node from LISTENER/FORWARD/FUNCTION/BANG/CLOCK point of view.

This way, it is possible to mute/play a stuff.

SYNTAX¶

The general syntactic scheme can be expressed like : 

((((IP(:(PORT)(','PORT)*)?)|((ALIAS)(','ALIAS)*)))?(':'|'+'|'-'|'!'|'#'|'?'|'='|'$'|'>'|'<')?('/'ADDR)+(','((((IP(:(PORT)(','PORT)*)?)|((ALIAS)(','ALIAS)*)))?)*

Multi Protocol Capabilities¶

While EzMojo is built on top of an extensible framework, it is possible to support other protocols for ingoing/outgoing communications.

  • Midi
  • DMX
  • Ubiquino (Arduino Firmata)
  • Ubiquino I2C
  • Ubiquino Servo
  • ThingM
  • XBee (Later on ...)
  • Living Colors
  • Pachube
  • Ableton Live (OSC mod)
  • Resolume Avenue (OSC mod)
  • Modul8 (OSC mod)

multi protocol support is done by specifying targeted protocol at the beginning of a command. None defined automatically use the default OSC 1.0 protocol.

It has to be mentioned that in a first time, only sending capabilities will operates. Due to a lack of time and specification it will be hard yet feasible to install multi protocol to OSC frame handlers in addition of the current development effort.
